
## 一、项目背景

当前跨境业务开户过程中，客户需要填写英文地址，但实际存在几个明显问题：

### 客户侧痛点

* 不熟悉国际地址书写规范
* 中文地址不会准确翻译
* 手机端输入复杂、耗时长
* 容易出现顺序错误、大小写错误、标点错误

### 业务侧痛点

* 客户经理需要反复指导
* 业务支持人工修正地址
* 审核效率低
* 地址真实性难判断

本质问题是：

> **客户提供的是“自然地址”，系统需要的是“标准地址”。**

中间缺少一个智能转换层。

---

# 二、方案目标

构建一个：

## **AI 地址智能标准化 Agent**

实现以下目标：

### 1. 自动翻译

中文地址 → 标准英文地址

### 2. 自动拆解

整段地址 → 结构化字段

### 3. 自动规范

统一国际地址格式

### 4. 自动校验

识别缺失 / 错误 / 异常地址

### 5. 自动辅助审核

降低后台人工工作量

补充约束：

* 当前阶段聚焦中国地址输入
* 输出目标为中国地址的国际英文格式
* 适用于跨境开户、KYC、合同资料、寄送信息等场景

# 四、核心能力模块

---

## 模块一：地址识别模块

判断输入类型：

* 中文地址
* 英文地址
* 拼音地址
* 中英混合
* 无效内容

例如：

输入：

```text
深圳市南山区科技园科苑路15号
```

识别结果：

```json
{
  "language": "zh",
  "type": "full_address"
}
```

---

## 模块二：地址解析模块

将非结构化地址拆成字段：

```json
{
  "country": "China",
  "province": "Guangdong",
  "city": "Shenzhen",
  "district": "Nanshan",
  "street": "Keyuan Road",
  "number": "15",
  "building": "Tencent Building",
  "room": "1203"
}
```

可识别：

### 国家层级

* Country

### 省级

* Province / State

### 城市

* City

### 区县

* District

### 街道

* Street

### 门牌

* Number

### 楼宇

* Building

---

## 模块三：智能翻译模块

将字段翻译为规范英文：

### 输入

```text
广东省深圳市南山区科苑路15号
```

### 输出

```text
No. 15 Keyuan Road
Nanshan District
Shenzhen
Guangdong
China
```

支持：

* 行政区标准翻译
* 固定地名保留
* 专有名词识别
* 常见缩写统一

例如：

* Road → Rd.
* Building → Bldg.
* Room → Rm.

---

## 模块四：格式标准化模块

当前阶段不做多国家模板切换，仅生成**中国地址的国际英文格式**。

模块目标不是简单翻译，而是把解析后的中国地址字段，按跨境表单、寄送面单、KYC 审核更容易接受的英文顺序重新组织。

---

### 中国地址格式（国际）

```text
Rm 1203, Tower A
Tencent Building
No. 15 Keyuan Rd.
Nanshan District
Shenzhen, Guangdong 518057
CHINA
```

---

### 推荐输出顺序

```text
第1行：房间 / 单元 / 楼层 / 座号
第2行：楼宇 / 园区 / 小区 / 公司名称
第3行：门牌号 + 街道名
第4行：街道办 / 镇 / 区县
第5行：城市, 省份 邮编
第6行：国家
```

---

### 标准化规则建议

* 优先使用“由细到粗”的国际阅读顺序，便于海外审核与物流系统识别
* 国家统一输出为 `CHINA`；如用于普通页面展示，也可显示为 `China`
* 邮编如已识别，建议放入“城市, 省份 邮编”这一行；如无法确认，不强行补齐
* 门牌统一为 `No. X`
* 房间统一为 `Rm XXX`
* 楼栋/塔楼统一为 `Tower A`、`Bldg. 2` 等英文形式
* 道路后缀统一缩写，例如 `Road -> Rd.`、`Street -> St.`、`Avenue -> Ave.`
* 区级行政区建议保留 `District`，例如 `Nanshan District`
* 专有楼宇名称若已有官方英文名，优先保留官方英文名；否则按拼音/约定译名输出
* 产业园、科技园、商务中心等名称应保留为独立一行，避免与街道行混写

---

### 输出字段建议

除文本地址外，建议同时输出一份标准结构，便于前端展示、回填表单、风控审核：

```json
{
  "formatted_address": [
    "Rm 1203, Tower A",
    "Tencent Building",
    "No. 15 Keyuan Rd.",
    "Nanshan District",
    "Shenzhen, Guangdong 518057",
    "CHINA"
  ],
  "postal_code": "518057",
  "country_code": "CN",
  "language": "en",
  "format_version": "CN_INTL_V1"
}
```

---

### 异常处理规则

* 缺失门牌号时，不伪造 `No.`，直接输出街道与风险提示
* 缺失楼宇名时，不强行生成楼宇行
* 同时存在中英文内容时，优先保留用户已填写且可信的官方英文专名
* 解析不确定时，允许输出多个候选，并要求用户确认
* 对“科技园/工业园/某某大厦”这类 POI 级信息，应与行政区信息分层输出，避免合并到同一行

---

## 模块五：智能校验模块

自动检查：

### 缺失字段

* 无国家
* 无城市
* 无门牌号

---

### 格式错误

例如：

```text
China Shenzhen No15
```

提示：

```text
建议补充详细街道信息
```

---

### 逻辑错误

例如：

```text
Tokyo, China
```

识别冲突：

* 国家与城市不匹配

---

### 风险评分

生成可信度：

```json
{
  "confidence_score": 87
}
```

规则：

* > 90 自动通过
* 70–90 提示确认
* <70 人工审核

---

### 建议拆分为三层校验

#### 第一层：规则校验

检查格式是否完整、字段是否合理：

* 是否包含国家 / 省 / 市 / 区 / 街道等核心字段
* 邮编是否为 6 位数字
* `province-city-district` 组合是否存在明显冲突
* 门牌号、楼栋号、房号是否符合常见表达习惯
* 英文输出是否符合 `CN_INTL_V1` 模板

#### 第二层：知识库校验

基于内置地址知识库做一致性判断：

* 行政区名称与标准英文映射是否一致
* 街道名称是否存在常见别名 / 旧称 / 简写
* 邮编与城市 / 区县是否大致匹配
* 楼宇、园区、商务中心是否属于已知高频地址词库

#### 第三层：联网验证

在用户授权或业务允许的前提下，调用外部地图 / 地址验证服务：

* 地址能否被地理编码成功
* 是否存在同名街道、多候选地址冲突
* 楼宇或 POI 是否可被外部平台识别
* 外部返回的结构化结果与 AI 结果是否一致

---

### 联网验证返回建议

```json
{
  "validation_level": "online_verified",
  "match_status": "partial_match",
  "provider": "amap",
  "provider_confidence": 0.82,
  "issues": [
    "postal_code_missing",
    "building_not_confirmed"
  ]
}
```

---

### 风险评分建议组成

建议不要只给单一分数，而是拆成可解释的多维评分：

* `parse_score`：解析可信度
* `translation_score`：翻译可信度
* `format_score`：格式规范度
* `knowledge_score`：知识库匹配度
* `online_verify_score`：联网验证匹配度

最终总分可按权重合成：

```text
total_score =
0.30 * parse_score +
0.15 * translation_score +
0.15 * format_score +
0.20 * knowledge_score +
0.20 * online_verify_score
```

这样后台审核人员可以知道“为什么低分”，而不是只看到一个黑盒结果。

## 方案二：高保真前端交互设计（Editorial / Tech-Luxury 风格）

为了突破传统 B2B 表单“枯燥、机械”的刻板印象，本方案提供了一套完整的 **“Editorial / Tech-Luxury（杂志排版与科技轻奢）”** 风格高保真前端设计。主色调采用品牌色 **`#435BCE`**，结合极简的冷灰白与深邃的暗夜蓝，形成强烈的视觉冲击与空间层次感。

### 1. 核心设计语言 (Design Aesthetic)

*   **排版组合 (Typography)**：摒弃常规系统字体。标题采用具有古典优雅气质的衬线体（`Playfair Display`），正文及数据展示采用现代利落的几何无衬线体（`Outfit`），形成“传统与未来”的戏剧性碰撞。
*   **空间构图 (Spatial Composition)**：打破传统的居中表单对齐，采用**非对称的双栏切割布局**。左栏（输入区）留白极其克制，右栏（解析区）沉浸且深邃。
*   **交互动效 (Motion)**：拒绝平庸的 Loading 圈。提交时触发“光刻扫描”动效；解析完成时，结构化字段采用阶梯式上浮（Staggered Fade-up）呈现，带来类似“高档票据打印”的仪式感。
*   **色彩矩阵 (Color Palette)**：
    *   主品牌色 (Primary)：`#435BCE`（用于核心按钮、底边框划线聚焦、关键高亮）
    *   背景空间 (Background)：`#F4F6FB`（冷灰白，用于外围空间与左侧面板）
    *   深色数据舱 (Dark Data Vault)：`#0B0E1A`（极暗蓝，用于右侧解析结果展示舱，带有微弱的噪点质感）

### 2. 界面结构详解

*   **左侧：沉浸式输入画布 (The Input Canvas)**
    *   **超大视觉引导**：“Address Intelligence.” 搭配极简的说明文案。
    *   **无边界输入框**：输入区域不再是传统的带边框矩形，而是一整块“无边界画布”。仅在输入框聚焦（Focus）时，底部会滑出 `#435BCE` 的锋利下划线。
*   **右侧：结构化数据舱 (The Data Vault)**
    *   默认状态为极暗蓝色的系统待机（System Standby）状态。
    *   解析后，字段被拆解为类似“数字航票”的数据卡片组合，包含核心置信度（Confidence Score）、细颗粒度拆解（Structured Entities）以及最终标准化输出（Standardized Output）。
    *   底部包含微型的校验状态指示灯（Format Rules: Pass / Geocode Match: High）。

## 组成模块

### LLM

负责：

* 理解自然语言地址
* 翻译
* 纠错

建议做成可切换的 Provider Adapter，而不是把业务逻辑直接绑定到某一家模型平台。

推荐配置项：

```env
LLM_PROVIDER=deepseek

CLAUDE_PROXY_BASE_URL=https://gaccode.com/claudecode
CLAUDE_PROXY_API_KEY=your_key
CLAUDE_PROXY_MODEL=your_claude_model

DEEPSEEK_BASE_URL=https://api.deepseek.com/v1
DEEPSEEK_API_KEY=your_key
DEEPSEEK_MODEL=deepseek-chat
```

接入原则：

* `Claude 中转站`：按其中转站实际接口协议接入，必须以其文档为准，不建议在业务层假设其 SDK 兼容方式
* `DeepSeek 官方`：官方文档明确支持 OpenAI 兼容调用，`https://api.deepseek.com` 与 `https://api.deepseek.com/v1` 均可，其中 `/v1` 仅用于兼容 OpenAI 风格路径
* 业务层统一只依赖内部抽象接口，例如 `normalizeAddress()`、`parseAddress()`、`validateAddress()`
* Prompt、输出 JSON Schema、重试机制、超时控制应在 Adapter 层统一管理

模型选型建议：

* `Claude` 更适合复杂地址理解、模糊纠错、多候选解释
* `DeepSeek` 更适合成本敏感、大批量调用、标准化输出
* 生产环境建议保留灰度切换能力，按成本、延迟、质量动态切换

# 九、建议新增：联网验证模块

---

如果这个系统只做“翻译 + 格式化”，已经可以提升填写体验；但如果希望它进一步服务于 KYC、合同资料审核、寄送、企业客户准入，那么**建议增加联网验证模块**。

原因不是让外部平台替代大模型，而是让大模型结果获得“外部世界证据”。

---

## 为什么值得加

联网验证可以解决仅靠 LLM 很难稳定解决的问题：

* 地址看起来合理，但现实中并不存在
* 街道存在，但门牌不确定
* 楼宇名是旧名、别名或口语表达
* 同一地址有多个英文写法，需要选更接近真实世界数据的一种
* 审核人员需要“可解释证据”，而不是只有模型主观判断

---

## 推荐的能力拆分

### 1. 地理编码验证

目标：判断地址是否可被地图服务识别。

输出：

* 是否命中
* 命中候选数
* 标准化返回地址
* 经纬度 / 行政区 / POI 信息

适合解决：

* 街道、区县、城市是否真实存在
* 地址是否过于模糊
* 是否存在多个近似候选

### 2. POI / 楼宇验证

目标：判断“某某大厦 / 科技园 / 商务中心 / 工业园”是否真实存在。

适合解决：

* 公司填写的是楼宇名而不是门牌
* 同一街道存在多个园区或楼宇
* 楼宇英文名是否有通用写法

### 3. 邮寄可达性 / 地址验证

目标：判断地址是否不仅格式正确，而且更接近“可投递、可触达”。

适合解决：

* 海外邮寄地址标签生成
* 对接物流、信函、纸质合同寄送
* 对地址真实性要求更高的业务流程

---

## API 选型建议

### 方案 A：第一阶段最小可用

接入中国本土地图能力即可：

* 高德开放平台 API
* 腾讯位置服务 API

适用场景：

* 中国地址识别
* 地址转坐标 / 坐标转地址
* POI 检索
* 地址补全

建议优先级：

* 若核心需求是“中国地址标准化 + 基础真实性校验”，优先高德或腾讯即可

### 方案 B：第二阶段企业增强

在地图能力之外，再加全球地址验证厂商：

* Loqate
* Smarty
* Experian Address Validation

适用场景：

* 需要更强的国际邮寄格式与可达性验证
* 需要企业级地址清洗、批量校验、可投递性验证
* 需要更强的审计与数据质量治理能力

建议判断标准：

* 若业务重点是“国内地址输入，输出英文给海外审核看”，地图服务通常够用
* 若业务重点是“海外寄送 / 国际邮政 / 全球地址治理”，应评估 Loqate / Smarty / Experian

## 联网验证的判定策略

建议不要简单做“查到就通过，查不到就拒绝”，而是采用分级判定：

* `strong_match`：街道、区县、城市、楼宇高度一致
* `partial_match`：街道与行政区一致，但门牌或楼宇未确认
* `weak_match`：只命中区县或街道，地址过于模糊
* `no_match`：无法匹配

对应处理：

* `strong_match`：可自动通过或高分通过
* `partial_match`：提示客户补充信息
* `weak_match`：提示重填或人工确认
* `no_match`：进入人工审核

## 合规与技术注意事项

* 中国地图平台通常返回中国本地坐标体系，若后续涉及地图展示，需要明确坐标体系处理方式
* 地址属于敏感业务数据的一部分，发送给第三方前应评估合规、脱敏、留痕与供应商协议
* 外部验证不应阻塞核心提交流程，建议做超时控制与降级兜底
* 需要缓存高频查询结果，降低成本与接口时延
* 应记录“模型结果”和“外部验证结果”的差异，持续反哺 Prompt 与知识库优化
